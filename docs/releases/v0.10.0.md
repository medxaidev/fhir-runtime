# fhir-runtime v0.10.0 Release Notes

**Release Date**: 2026-03-18
**Stage**: STAGE-7 — Profile Slicing & UI Utility API
**Breaking Changes**: None (additive only)
**Dependency**: fhir-definition@0.6.0 (unchanged)

---

## Highlights

v0.10.0 addresses the most critical gaps identified by the `fhir-runtime-tools` integration analysis:
- **Slicing support**: `buildCanonicalProfile()` now preserves slice definitions, and new APIs provide discriminator matching, instance counting, and skeleton generation.
- **inferComplexType fix**: Improved ContactPoint vs Identifier disambiguation eliminates false-positive TYPE_MISMATCH validation errors.
- **Choice Type utilities**: Standard functions for detecting, parsing, and resolving FHIR choice types (`[x]` elements).
- **BackboneElement utilities**: Standard functions for detecting backbone elements and retrieving child elements.

These additions reduce application-layer self-implementation by ~520 lines and provide standardized FHIR semantic APIs that benefit all fhir-runtime consumers.

## New Features

### 7.1 — Profile Slicing Preservation

`buildCanonicalProfile()` now uses a two-pass algorithm:
- **Pass 1**: Processes non-slice elements (id without `:`) into `elements` Map
- **Pass 2**: Extracts slice elements into `profile.slicing` Map

New types: `SlicedElement`, `SliceDefinition` on `CanonicalProfile`.

```typescript
const profile = buildCanonicalProfile(usCoreObservationSD);
const catSlicing = profile.slicing?.get('Observation.category');
// catSlicing.slices[0].sliceName === 'VSCat'
// catSlicing.discriminators[0].type === 'pattern'
```

### 7.2 — Slicing API

New functions in `src/profile/slicing-utils.ts`:

| Function | Description |
|----------|-------------|
| `matchSlice(instance, slicedElement)` | Match instance to slice by discriminator |
| `countSliceInstances(items, slicedElement)` | Count instances per slice |
| `generateSliceSkeleton(slice)` | Generate pre-filled skeleton |
| `isExtensionSlicing(basePath)` | Check if path is extension slicing |

### 7.3 — inferComplexType Fix

Improved disambiguation logic for ContactPoint vs Identifier:
- Checks Identifier-specific fields (`type`, `assigner`) first
- `mobile` use value is ContactPoint-only
- URI-format `system` with overlapping `use` values resolves to Identifier

### 7.4 — Choice Type Utilities

New functions in `src/profile/choice-type-utils.ts`:

| Function | Description |
|----------|-------------|
| `isChoiceType(element)` | Detect `[x]` elements |
| `getChoiceBaseName(path)` | `"value[x]"` → `"value"` |
| `buildChoiceJsonKey(base, type)` | `("value", "Quantity")` → `"valueQuantity"` |
| `parseChoiceJsonKey(key, base)` | `"valueQuantity"` → `"Quantity"` |
| `resolveActiveChoiceType(el, resource)` | Find active variant |
| `resolveChoiceFromJsonKey(key, elements)` | Reverse-resolve from JSON key |

New type: `ChoiceTypeResolution`.

### 7.5 — BackboneElement Utilities

New functions in `src/profile/backbone-utils.ts`:

| Function | Description |
|----------|-------------|
| `isBackboneElement(element)` | Detect backbone elements |
| `isArrayElement(element)` | Detect array elements (max > 1) |
| `getBackboneChildren(path, profile)` | Get direct children (filters id/extension/modifierExtension) |

## Compatibility

- **Backward compatible**: All changes are additive
- `CanonicalProfile.slicing` is optional — existing code unaffected
- `buildCanonicalProfile()` behavior fix: base elements no longer overwritten by slice elements
- `inferComplexType()` fix: reduces false positives, does not affect correct inferences

## Migration from v0.9.0

No code changes required. To use new features:

```typescript
import {
  // Slicing
  matchSlice, countSliceInstances, generateSliceSkeleton, isExtensionSlicing,
  // Choice Type
  isChoiceType, getChoiceBaseName, buildChoiceJsonKey, resolveActiveChoiceType,
  // BackboneElement
  isBackboneElement, isArrayElement, getBackboneChildren,
  // Types
  type SlicedElement, type SliceDefinition, type ChoiceTypeResolution,
} from 'fhir-runtime';
```
