# fhir-runtime v0.8.0 Release Notes

**Release Date:** 2026-03-12  
**Package:** `fhir-runtime@0.8.0`  
**FHIR Version:** R4 (4.0.1)  
**Previous Version:** v0.7.2  
**Stage:** STAGE-6 Complete

---

## Overview

v0.8.0 completes **STAGE-6: fhir-definition Integration**, introducing the `definition` module (12th module) that integrates `fhir-definition` v0.4.0 as a runtime dependency. This release provides a unified `DefinitionProvider` interface for consuming FHIR definitions (StructureDefinition, ValueSet, CodeSystem, SearchParameter) and introduces the `createRuntime()` factory function for simplified initialization.

**Key Achievement**: fhir-runtime can now directly consume definitions from `fhir-definition`'s `InMemoryDefinitionRegistry`, enabling seamless IG package loading and multi-source definition management.

## What's New

### 1. New Module: `definition`

**Source**: `src/definition/` (6 files)  
**Exports**: 15 (11 type + 4 value)

#### 1.1 Core Interfaces (Re-exported from fhir-definition)

- **`DefinitionProvider`** ‚Äî Core interface for consuming FHIR definitions
  - `getStructureDefinition(url: string): StructureDefinition | undefined`
  - `getValueSet(url: string): ValueSet | undefined`
  - `getCodeSystem(url: string): CodeSystem | undefined`
  - `getSearchParameters(resourceType: string): SearchParameter[]`

- **`DefinitionRegistry`** ‚Äî Interface for definition registry
- **`InMemoryDefinitionRegistry`** ‚Äî In-memory implementation (from fhir-definition)
- **`RegistryStatistics`** ‚Äî Statistics for loaded definitions

#### 1.2 Adapters & Implementations

- **`DefinitionBridge`** ‚Äî Adapter composing `FhirContext` + optional VS/CS/SP registries into unified `DefinitionProvider`
- **`NoOpDefinitionProvider`** ‚Äî Default no-op implementation returning undefined/empty for all queries
- **`DefinitionProviderLoader`** ‚Äî `StructureDefinitionLoader` adapter bridging `DefinitionProvider` into `FhirContext`'s loader pipeline

#### 1.3 Runtime Factory

- **`createRuntime(options?: RuntimeOptions): Promise<FhirRuntimeInstance>`**
  - One-step creation of fully configured runtime instance
  - Three usage patterns:
    1. With external `DefinitionProvider` (from fhir-definition) ‚Äî **recommended**
    2. With `FhirContext` (legacy/standalone mode)
    3. Bare minimum (auto-creates everything with defaults)

- **`FhirRuntimeInstance`** ‚Äî Unified runtime object exposing:
  - `validate(resource, profileUrl): Promise<ValidationResult>`
  - `getSearchParameters(resourceType): SearchParameter[]`
  - `extractSearchValues(resource, searchParam): SearchIndexEntry`

## Usage Examples

### Pattern 1: With fhir-definition (Recommended)

```typescript
import { createRuntime } from 'fhir-runtime';
import { InMemoryDefinitionRegistry, loadFromDirectory } from 'fhir-definition';

// 1. Load definitions from fhir-definition
const registry = new InMemoryDefinitionRegistry();
await loadFromDirectory('./fhir-packages/hl7.fhir.us.core', registry);

// 2. Create runtime with definitions
const runtime = await createRuntime({ definitions: registry });

// 3. Validate US Core resource
const result = await runtime.validate(
  usPatient,
  'http://hl7.org/fhir/us/core/StructureDefinition/us-core-patient'
);

// 4. Get search parameters
const sps = runtime.getSearchParameters('Patient');
```

### Pattern 2: Bare Minimum

```typescript
import { createRuntime } from 'fhir-runtime';

// Auto-loads R4 core definitions
const runtime = await createRuntime();

// Validate against base FHIR profiles
const result = await runtime.validate(
  patient,
  'http://hl7.org/fhir/StructureDefinition/Patient'
);
```

### Pattern 3: With Existing FhirContext

```typescript
import { createRuntime, FhirContextImpl, MemoryLoader } from 'fhir-runtime';

const context = new FhirContextImpl({ loaders: [new MemoryLoader(new Map())] });
await context.preloadCoreDefinitions();

const runtime = await createRuntime({ context });
```

## Important Notes

### Dependency Change

- **v0.7.x**: Zero runtime dependencies
- **v0.8.0**: Single dependency: `fhir-definition@0.4.0`

This is an intentional architectural decision to enable powerful definition management capabilities while maintaining a minimal dependency footprint.

### FhirContextImpl with Empty Loaders

```typescript
// ‚ùå Throws error (CompositeLoader requires ‚â•1 loader)
new FhirContextImpl({ loaders: [] })

// ‚úÖ Correct approach
new FhirContextImpl({ loaders: [new MemoryLoader(new Map())] })

// ‚úÖ Or use createRuntime()
await createRuntime()
```

## Design Decisions

### ADR: Direct Dependency on fhir-definition

**Decision**: Import types directly from `fhir-definition` package, not redefine them.

**Rationale**:
- Avoids type duplication
- Ensures type compatibility
- Simplifies maintenance
- Enables seamless integration

**Trade-off**: Adds one runtime dependency, but provides significant value (IG package management, multi-source definitions, dependency resolution).
