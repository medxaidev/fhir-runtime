# fhir-runtime v0.11.0 Release Notes

**Release Date**: 2026-03-18
**Scope**: IG Extraction API (REQ-13)
**Breaking Changes**: None (additive only)
**Dependency**: fhir-definition@0.6.0 (unchanged)

---

## Highlights

v0.11.0 implements the **IG Extraction API** requested by Phase-fhir-server-004 (REQ-13). These utilities extract structured data from FHIR conformance resources for database indexing, enabling standardized IG import pipelines.

The three new functions eliminate ~100 lines of duplicated extraction logic that would otherwise be needed in fhir-server and fhir-persistence, while leveraging fhir-runtime's existing type system.

---

## New Features

### R1 — extractSDDependencies

Extract all direct type dependencies from a StructureDefinition's snapshot.

```typescript
import { extractSDDependencies } from 'fhir-runtime';

const deps = extractSDDependencies(usCorePatientSD);
// → ['Address', 'Attachment', 'CodeableConcept', 'ContactPoint',
//    'Extension', 'HumanName', 'Identifier', 'Reference',
//    'http://hl7.org/fhir/us/core/StructureDefinition/us-core-race', ...]
```

**Features**:
- Collects `type.code`, `type.profile[]`, `type.targetProfile[]`
- Excludes FHIR primitive types by default (configurable via `includePrimitives`)
- Excludes self URL
- Returns sorted, de-duplicated array

**Location**: `src/profile/sd-dependency-extractor.ts`

### R2 — extractElementIndexRows

Extract element index rows from a StructureDefinition snapshot for database table population.

```typescript
import { extractElementIndexRows } from 'fhir-runtime';
import type { ElementIndexRow } from 'fhir-runtime';

const rows: ElementIndexRow[] = extractElementIndexRows(patientSD);
// rows[0] = {
//   id: 'Patient:Patient',
//   structureId: 'Patient',
//   path: 'Patient',
//   min: 0, max: '*',
//   typeCodes: [],
//   isSlice: false, isExtension: false, mustSupport: false,
// }
```

**Features**:
- Maps each snapshot element to an `ElementIndexRow`
- Detects slices (`sliceName`), extensions, mustSupport
- Extracts binding valueSet
- Handles choice types (multiple type codes)
- Uses `sd.id` as structureId, falls back to `sd.url`

**Location**: `src/profile/element-index-extractor.ts`

### R3 — flattenConceptHierarchy

Flatten a CodeSystem's nested concept hierarchy into parent-child rows.

```typescript
import { flattenConceptHierarchy } from 'fhir-runtime';
import type { ConceptRow } from 'fhir-runtime';

const rows: ConceptRow[] = flattenConceptHierarchy(snomedSubsetCS);
// rows = [
//   { code: 'clinical-finding', parentCode: null, level: 0, ... },
//   { code: 'disorder', parentCode: 'clinical-finding', level: 1, ... },
//   ...
// ]
```

**Features**:
- Recursively walks concept tree
- Records `parentCode` and `level` for each concept
- Accepts both raw FHIR CodeSystem JSON (`concept[]`) and runtime `CodeSystemDefinition` (`concepts[]`/`children[]`)
- Generates unique IDs as `{url}:{code}`

**Location**: `src/terminology/concept-hierarchy-extractor.ts`

## New Types

```typescript
interface ElementIndexRow {
  id: string;
  structureId: string;
  path: string;
  min?: number;
  max?: string;
  typeCodes: string[];
  isSlice: boolean;
  sliceName?: string;
  isExtension: boolean;
  bindingValueSet?: string;
  mustSupport: boolean;
}
```

### ConceptRow

```typescript
interface ConceptRow {
  id: string;
  codeSystemUrl: string;
  codeSystemVersion?: string;
  code: string;
  display?: string;
  parentCode: string | null;
  level: number;
}
```

## Migration from v0.10.0

No code changes required. To use new features:

```typescript
import {
  extractSDDependencies,
  extractElementIndexRows,
  flattenConceptHierarchy,
  type ElementIndexRow,
  type ConceptRow,
} from 'fhir-runtime';
```

## Relationship to fhir-server / fhir-persistence

These APIs are designed as upstream utilities for the IG import pipeline defined in Phase-fhir-server-004:

- `extractSDDependencies` → used by `GET /_ig/{igId}/structure/{sdId}` to return dependency lists
- `extractElementIndexRows` → used by `IGImportOrchestrator` to populate `structure_element_index` table
- `flattenConceptHierarchy` → used by `IGImportOrchestrator` to populate `code_system_concept` table

See `devdocs/all/stage/13-REQ-fhir-runtime-ig-extraction-api.md` for the full requirements specification.
